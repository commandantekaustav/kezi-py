Phase 12: Passing the Intel

Professor's Briefing

An SOP that does the same exact thing every time is limited. A truly useful function needs to accept intel—parameters for the current job—and then report back with the result. Today, we upgrade our functions to accept **arguments** (input) and to **return** values (output).

The Blueprint: Arguments and Return Values

An **argument** is a piece of information you pass *into* a function. A **return value** is the result the function sends *back out*.

Code Workbench


# This function now accepts one argument: 'name'.
# 'name' is a placeholder variable that holds whatever value we pass in.
def create_greeting(name):
    # It creates a personalized greeting string.
    greeting = f"Welcome, operative {name}. The mission is a go."
    # The 'return' keyword sends the value of 'greeting' back out.
    return greeting

# This function checks a score and returns a boolean value.
def is_hireable(score):
    if score >= 85:
        return True
    else:
        return False

# Now we call the functions, passing in our intel.
candidate_1_name = "Priya Sharma"
candidate_1_greeting = create_greeting(candidate_1_name)
print(candidate_1_greeting)

candidate_2_name = "David Chen"
print(create_greeting(candidate_2_name)) # We can also print the returned value directly.

# Let's check their scores.
candidate_1_score = 92
candidate_2_score = 78

print(f"Is {candidate_1_name} hireable? {is_hireable(candidate_1_score)}") # Prints True
print(f"Is {candidate_2_name} hireable? {is_hireable(candidate_2_score)}") # Prints False

Why This Matters To You

This is the essence of modular, professional code. Developers build small, smart functions that do one thing well: they take in data, process it, and return a result. Complex programs are just layers and layers of these smart functions calling each other, passing intel up and down the chain of command.
